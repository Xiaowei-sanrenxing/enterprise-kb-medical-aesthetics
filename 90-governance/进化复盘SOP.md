# 🔄 进化复盘 SOP · 把"踩过的坑"变成"库的免疫力"

> 这是让 〔机构名〕 这套库**自我进化**的操作手册。定期跑一次（建议每周或每月一次，15-30 分钟）。
> 不跑 = 库是死的；跑了 = 库越用越准、越用越值钱。

## 谁来跑
知识库负责人 + AI。用 Claude Code 的，直接一句话："**按进化复盘SOP，复盘这周的进化日志**"，AI 自动走下面五步。

## 五步法

1. **收集** —— 打开 [`进化日志.md`](进化日志.md)，取出所有「回流状态 = 待复盘」的行。

2. **聚类** —— 把同类问题归一堆：
   - 同一条红线被反复踩？　- 同一类客户问题反复出现？　- 同一处任务卡指令反复被误解？

3. **提炼** —— 每一堆问一句："**库该怎么改，才能让下次 agent 自动避开 / 答对？**"
   | 日志里的现象 | 库该怎么进化 |
   |---|---|
   | 反复踩同一红线 | 写进对应 `行业合规默认.md` 或任务卡的红线清单 |
   | 反复被问同一问题 | 补一份 `context/` 资产，或加进话术 |
   | 任务卡指令有歧义 | 把指令改清楚 |
   | 出现库没覆盖的新坑 | 加一道 `evals.md` 验收题守住它 |

4. **回流** —— 真的去改对应工作流的 `context/` / `_task-card.md` / `evals.md`（带 6 元数据，默认 `status: 草稿`，负责人确认后转 `定版`）。

5. **标记** —— 把日志里这些行的「回流状态」改成 `已回流（→ 改了什么）`。

## 📊 进化账本（让"越用越值钱"看得见 → 这是续费的硬证据）

每轮复盘后，在日志底部记一笔：
> `2026-0X-XX 复盘：新增 X 条红线、X 份 context、X 道 evals；解决高频坑 X 个。`

积累三个月，你就能拿出一句话向老板/客户证明：
> **"这套库三个月里自动消化了 N 个踩坑点、长出了 M 条新标准——它在替我们把老师傅的经验一点点固化下来。"**
